# ApplyFlow — Database Tables & Relationships Explanation

This document explains the purpose, key fields, and relationships of each table in ApplyFlow. It is designed to be **read independently** and give a clear understanding of how to navigate the database.

---

## 1. Users Table (`users`)

**Purpose:**  
Stores basic information about every person using ApplyFlow.

**Key fields:**

- `id` → Primary Key (UUID)  
- `email` → Unique login  
- `created_at` → Account creation timestamp  

**Notes:**  

- Every table that needs ownership links back to `users.id`.

---

## 2. Applications Table (`applications`)

**Purpose:**  
Represents each job the user wants to track. This is the **core of the workflow**.

**Key fields:**

- `id` → PK  
- `user_id` → FK → `users.id` (owner of the application)  
- Job details: `company_name`, `role_title`, `location`, `salary_min`/`salary_max`, `currency`, `job_url`, `job_description`  
- `current_stage` → enum: tracks application stage (`wishlist → applied → interview → offer → rejected`)  
- `resume_version_id` → FK → `resume_versions.id` (resume used for this application)  

**Notes:**  

- Parent table for notes, reminders, AI analyses, and interview questions.  
- Changes in `current_stage` are tracked via `application_notes`.

---

## 3. Resumes & Versions

### Parent Table (`resumes`)

- Groups related resumes under a single **“base resume”**  
- `id` → PK  
- `user_id` → FK → `users.id`  
- `title` → e.g., “Software Engineer Resume”

### Version Table (`resume_versions`)

- Tracks individual uploads or tailored edits  
- `id` → PK  
- `resume_id` → FK → `resumes.id`  
- `file_url` → PDF storage location  
- `parsed_content` → Extracted text for AI analysis  
- `created_at` → Timestamp  

**Key Principle:**  

- Versioning is **mandatory**. Never overwrite originals.  
- AI analyses always tie to a specific `resume_version`.

---

## 4. AI Analyses (`ai_analyses`)

**Purpose:**  
Stores AI-generated insights for resumes and applications.

**Key fields:**

- `id` → PK  
- `user_id` → FK → `users.id`  
- `application_id` → FK → `applications.id`  
- `resume_version_id` → FK → `resume_versions.id`  
- `analysis_type` → enum: `resume_review`, `job_match_score`, `interview_prep`, `followup_draft`  
- `result_json` → Structured JSON output from AI  

**Notes:**  

- Each AI analysis is tied to the **resume version used at the time**.  
- Deleting an application cascades to its AI analyses to keep the database clean.

---

## 5. Interview Questions (`interview_questions`)

**Purpose:**  
Tracks questions generated by AI for interview preparation.

**Key fields:**

- `application_id` → FK → `applications.id`  
- `ai_analysis_id` → optional FK → `ai_analyses.id`  
- `question_text`, `category` (technical/behavioral), `tips`  
- `is_practiced` → boolean  
- `user_notes` → text  

**Notes:**  

- Each question belongs to one application.  
- Status (`is_practiced`) and notes are **per-user, per-question**.

---

## 6. Notes & Reminders

### Application Notes (`application_notes`)

- Tracks status changes, general notes, or follow-ups  
- `metadata` JSONB stores structured info (like stage changes)  

### Follow-Up Reminders (`follow_up_reminders`)

- Simple reminders linked to applications  
- Fields: `title`, `due_date`, `status`  

**Notes:**  

- Both tables **cascade delete** when their application is removed.

---

## 7. Relationships Summary

| From | To | Relationship | Notes |
|------|----|-------------|-------|
| `applications.user_id` | `users.id` | One-to-Many | Each user can have multiple applications |
| `applications.resume_version_id` | `resume_versions.id` | One-to-One | Which resume was used for this job |
| `resume_versions.resume_id` | `resumes.id` | One-to-Many | Versions grouped under parent resume |
| `ai_analyses.application_id` | `applications.id` | One-to-Many | Each application can have multiple AI insights |
| `ai_analyses.resume_version_id` | `resume_versions.id` | One-to-One | Links analysis to the exact resume used |
| `interview_questions.application_id` | `applications.id` | One-to-Many | Questions belong to an application |
| `application_notes.application_id` | `applications.id` | One-to-Many | Tracks all notes/changes per application |
| `follow_up_reminders.application_id` | `applications.id` | One-to-Many | Tracks reminders per application |

**Cascading rules:**

- Deleting an application removes **notes, reminders, AI analyses, and interview questions** automatically.  
- Resumes are versioned; deleting a version does **not** destroy the parent resume or other versions.

---

## 8. How to Read the Diagram Quickly

- Look for the **central hub** → `applications`. Everything flows through it.  
- Identify **ownership** → `user_id`. Every table ties to a user.  
- Check **versioning** → `resumes → resume_versions`. AI always works with a specific version.  
- AI & prep → `ai_analyses` & `interview_questions`. Analyses connect resumes to applications. Questions connect analyses back to applications.  
- Support tables → `application_notes` & `follow_up_reminders`. Extra info, no AI dependence, purely tracking & accountability.
